一个像素的一生

Veröffentlicht am 2018-10-02

大体流程

web content -> magic（渲染） -> pixels

web content

web content主要包含HTML，CSS，JS。

  1. HTML（Hyper-Text Markup Language，页面结构与内容）
  2. CSS（Cascading Style Sheets，页面样式）
  3. JS（JavaScript，负责页面界面结构与内容和页面样式的更新）

其中在Chromium中用于渲染的引擎被称为 Blink 。渲染需要调用图像处理的底层API。其统一标准为openGL。目前正在开发Vulkan进行统一化。底层API不能读懂HTML和CSS，只能做简单的图像绘制。

生命周期

整个流程是要将web content转化为对应的图像处理API，在这个过程中，为了更好地将已经渲染的图像 更新，需要设计数据结构帮助我们更新页面的结构、内容和页面样式。这些更新包括JavaScript API，用户在，输入框的输入，异步加载，动画，卷轴移动，页面缩放等。

初始渲染

parse

style

CSS是HTML的样式，所以要依附到DOM对象上。这个依附的过程，也就是 CSS选择 的过程。但是由于HTML的树状结构，CSS有时需要写的十分复杂以去高效的匹配到相对应的DOM元素。

在CSS解析时，解析器会将每一个选择器所选择的CSS属性名和属性值保存，作为map。
CSS属性名是由C++进行生成的，该C++文件在构建时由python脚本自动生成。

下一步，称为重计算（recalc），对于所有产生的属性，我们会计算它们的叠加和。作为每一个DOM元素的每一个属性的值，这个值也称为计算值。也就是最终渲染的结构。

这个属性值我们可以用Chrome的API或者是JavaScript DOM API均可以得到。

layout

通过上面获得的计算属性，我们就可以确定每一个元素在视图上占据的确定位置。

这里举个一个简单的例子。就是每一个元素是由上向下一次排列的，每一个元素的高度由字体的大小和间距所决定。

但是，实际情况往往比较复杂。比如一个元素的内容超出了边界，那么该元素会呈现可滚动的特性，这时页面需要实时计算显示的区域。
再比如表格布局、浮动、文本分列、flex布局、writing-mode等属性，都会带来布局的复杂程度。
所以，我们需要一个更加完善的数据结构去存储这样一些状态，使得每一次迭代都能够变得高效和纯粹。

DOM节点基本同layout对象一一对应，有些许不同。例如，不是每一个DOM元素都对应有一个Layout Object，比如对于一个display属性设置为none的元素；同时也不是每一个Layout Object都对应一个DOM元素，比如伪元素。

这里引入新的数据结构，是对之前重计算所得到的Render Tree的进一步封装，将之前的复杂情况进一步考虑，得到最终每一个元素在视图上的最终为止，即 Layout Tree。

基于Layout Tree，我们可以处理overflow等一系列复杂情况。但是还有一个问题是，这种数据结构没有将输入的计算属性和输出的视图位置分开，所以这里提及了一个正在开发的新项目名为LayoutNG就是为了解决这个问题。

print

得到上个步骤的Layout对象也就意味着我们可以真正的绘制像素了。但注意，这里绘制也仅仅是语义上的，并没有落实到屏幕上。我们会再一次将Layout对象转化为一个个矩形和其对应的颜色，并且按其 堆栈 的形式显示，并非是DOM的出现顺序。并且每一次渲染都是根据某一种属性，比如PPT中简化的集中，背景、浮动、前景，轮廓。

raster

在得到元素最终在视图的显示信息，我们就需要真正的进行预渲染。在这一步，我们会将屏幕上每一个像素点的颜色生成一个32位的二进制码，分别表示4种颜色，同时我们可以利用GPU进行渲染加速。这底层使用的是Google自主开发的skia进行图像处理渲染，通过最底层的OpenGL。同时skia作为一个单独的项目，也支持了其他的大型项目，比如安卓系统。

gpu

最后需要注意的是真正通过skia调用的openGL API会通过CommandBuffer调用GPU进程，进行真实的渲染。也就是说，真实的渲染是独立出去的，当这一部分进程宕机时我们可以快速的重启。
